# Setup

This guide is for a maintainer who wants a first successful local run. It assumes you already know how to operate a Matrix homeserver or at least register an appservice on one.

## Prerequisites

- Go 1.24+.
- A Matrix homeserver where you can register an appservice.
- A Microsoft Teams consumer account that can sign in at `https://teams.live.com/v2`.
- A Matrix client or provisioning UI that can drive mautrix `bridgev2` login flows.
- SQLite for quick local testing, or Postgres for longer-lived deployments.

Important constraints:

- This repo currently targets `teams.live.com` consumer endpoints.
- Login is delegated-user auth only. There is no service-account or tenant-wide bootstrap path.
- The bridge itself does not create homeserver-side appservice registrations for you; it only generates the registration file.

## Build

From the repo root:

```sh
./build.sh
```

If you prefer plain Go:

```sh
go build -o mautrix-teams ./cmd/mautrix-teams
```

## Create A Starting Config

Use the documented example in [config.example.yaml](../config.example.yaml) as your starting point:

```sh
cp config.example.yaml config.yaml
```

If you want the full autogenerated mautrix config instead, run this in an empty directory or after moving any existing `config.yaml` out of the way:

```sh
./mautrix-teams -e
```

Recommended local defaults:

- `database.type: sqlite3-fk-wal`
- `database.uri: file:mautrix-teams.db?_txlock=immediate`
- `homeserver.software: standard` unless you are intentionally using Hungryserv/WebSocket mode
- `homeserver.websocket: false` for ordinary Synapse/Dendrite-style HTTP appservice operation

## Fill In Required Config

At minimum, review these sections:

- `homeserver`: Matrix base URL and server name.
- `appservice`: listener address, bot username, and registration tokens.
- `database`: SQLite or Postgres connection.
- `bridge.permissions`: which Matrix users may use the bridge.
- `provisioning`: secret for any provisioning/API-based login flow.
- `network.client_id`: optional override for Teams web app OAuth client ID. Leave empty unless Microsoft changes the default Teams client ID and login extraction starts failing.

## Generate The Appservice Registration

Once `config.yaml` is in place:

```sh
./mautrix-teams -g -c config.yaml -r registration.yaml
```

Register `registration.yaml` with your homeserver using that homeserver's normal appservice registration process, then restart or reload the homeserver as required.

## Run The Bridge

```sh
./mautrix-teams -c config.yaml
```

Expected first-run behavior:

- The bridge starts its Matrix appservice connection.
- Teams-specific DB tables are created or upgraded.
- The appservice bot becomes available on Matrix.
- No Teams traffic flows until a user completes the Teams login flow.

## Complete A Teams Login

The connector exposes a cookie/webview-based login flow named `teams.live.com (in-app browser)`.

What happens during login:

1. The user signs into `https://teams.live.com/v2` in an embedded browser.
2. The bridge extracts Teams MSAL localStorage from that session.
3. It derives a refresh token, then acquires a Teams `skypetoken`.
4. It also tries to acquire a Graph token for attachment support.
5. The user login is stored in the bridge database and background sync starts.

Practical note:

- Beeper already knows how to drive this flow.
- Outside Beeper, you need some bridgev2-compatible provisioning UI or custom integration that can submit the cookie/localStorage payload expected by the login process.

## Verify It Works

Use this checklist:

1. The bridge process stays up and logs a successful Matrix connection.
2. The user login reaches a connected state instead of `bad_credentials`.
3. A Teams chat appears as a Matrix portal room after thread discovery.
4. Sending a Matrix text message produces a new Teams message.
5. A Teams reply in that chat is polled back into Matrix within the polling window.

Useful signals in logs:

- Login started / waiting for localStorage submission.
- Graph token present or absent after login extraction.
- Thread discovery refresh succeeded.
- Per-thread polling continues without repeated 4xx or token errors.

## Common Setup Failures

### Login never completes

Likely causes:

- Your client/provisioning flow does not support the `bridgev2` cookie/webview login contract.
- Teams localStorage changed and the extractor no longer finds MSAL keys.

What to check:

- Bridge logs for `Teams webview login still waiting for cookie submission`.
- Bridge logs for `FI.MAU.TEAMS_INVALID_STORAGE` or missing access token errors.

### `bad_credentials` immediately after login

Likely causes:

- Refresh token was not extracted correctly.
- The default Teams OAuth client ID no longer matches the current web app.
- Microsoft changed the Skype token exchange behavior.

What to check:

- Re-run login and inspect logs around token extraction.
- If Microsoft changed the Teams web client ID, set `network.client_id` explicitly and try again.

### 401 / 403 from Teams APIs

Likely causes:

- The stored refresh token can no longer mint a valid `skypetoken`.
- The Teams account is not valid for the consumer endpoints this bridge uses.

Recovery:

- Re-login the user.
- Confirm the account works on `teams.live.com`, not only an enterprise Teams tenant flow.

### Attachment bridging fails while text works

Likely causes:

- Graph token refresh failed.
- The inbound Teams message did not include a usable Drive item ID.

Recovery:

- Re-login to refresh delegated Graph consent.
- Check whether messages fall back to text links instead of full file reupload. That is expected degradation, not a total bridge failure.

### Matrix side never connects

Likely causes:

- Homeserver URL or domain mismatch.
- Appservice registration not installed or tokens mismatched.
- WebSocket mode enabled against a standard homeserver.

Recovery:

- Double-check `homeserver.address`, `homeserver.domain`, `appservice.as_token`, `appservice.hs_token`.
- For normal homeservers, disable `homeserver.websocket`.
